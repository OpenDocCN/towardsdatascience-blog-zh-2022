# 良好的数据公民身份不起作用

> 原文：<https://towardsdatascience.com/good-data-citizenship-doesnt-work-265f13a37fa5>

## 我们可以也应该要求每个人都成为良好的数据公民。但是我们也必须成为优秀的领导者。

从小学开始，我们都被教育要成为好公民。我们从公民学老师那里了解到，如果我们不投票，不参与地方政治，民主是行不通的；我们从社会领袖那里听到，我们必须大声疾呼，让人们听到我们的声音；民选官员告诉我们，我们不应该问国家能为我们做什么，而应该问我们能为国家做什么。

这是一个强有力的信息。但是不完整。光靠公民无法创造繁荣的社会；当选领导人也必须尽自己的职责。他们必须起草法案，通过法律，协商妥协，并建立公平有效的机构来执行人民的意愿。正如人们不能把民主的责任推给政府，政府也不能把领导的责任推给人民。

数据组织——越来越多地围绕“数据民主化”和“公民分析师”这样的术语来构建他们的目标——需要记住同样的事情。随着数据的重要性和范围的增长，数据团队试图将责任外包给他们的非数据同事，要求他们中的更多人管理、分析和管理组织的分析资产。总的来说，这种努力是有意义的:数据团队太小，无法独自完成这项工作，正如我们的政治进程一样，公民的更多参与有助于建设一个更有活力的社会。

但是我们应该小心不要走得太远。组织“选举”我们成为公司数据的主要管理者，并建立既能让他们有效参与又能满足他们需求的机构。我们可以也应该要求每个人都成为良好的数据公民。但我们也必须成为优秀的领导者，尤其是在与管理和组织数据资产相关的事务上。

# 从全程服务到自助服务

几十年前，当数据主要用于向高管报告一些业务 KPI 时，数据团队(或者更常见的是 IT 团队)可以独立完成大部分工作。不需要数据公民，就像不需要餐馆里的公民一样:几个人下几份订单，厨房就能很好地处理。

近年来，这一切都改变了。顾客的数量大幅增长，从高管和少数运营领导到公司的每个人。订单的种类也呈爆炸式增长。数据不仅用于 KPI，还用于回答各种主题的复杂问题，从产品和工程到运营、客户支持和执行决策。每个订单的复杂性也增加了。过去存在于几个中央系统中的数据现在分散在数十个(如果不是数百个)第三方 SaaS 应用程序中。根据一些消息来源的说法，如今组织使用的 SaaS 产品是五年前的十倍多[1]。

作为一个数据行业，我们通过[建立自助餐](https://thesequel.substack.com/p/the-endless-data-buffet)来应对这种巨大的需求增长。使用自助工具，人们可以创建自己的分析，顾客可以根据数据团队提供的一些基本成分组装自己的订单。这家新餐馆可以为比以前多得多的人服务，极大地扩展了组织中每个人对数据的访问。

但是这产生了一个新问题:人们如何知道他们可以信任他们看到的数据？把自助餐的比喻延伸一下，人们怎么知道他们在街角的车站找到的食物，那种感觉整晚都没人注意的食物，仍然可以安全食用？

很多时候，事实并非如此。人们会出现在一个会议上讨论一个业务问题，不同的人会带来不匹配的号码。很多时候，数据会减缓决策，而不是加快决策，因为人们花在争论数据是否正确上的时间比花在讨论如何处理数据上的时间要多。通常，数据团队会更新核心数据集或 KPI，旧版本会持续几个月。

诸如此类的事件侵蚀了人们对数据和数据团队的信任。它教导人们不要参与数据民主，而是完全靠自己做决定，通常根本不使用数据。

# 打造更好的菜单

为了解决这个问题，许多数据团队转向数据文档。这个想法很简单:创建一个菜单，解释哪些数据是可用的，以及它们的含义。虽然这种文档可以采取许多不同的形式——内部维基、第三方产品、聊天机器人、谷歌表单——为了让自助服务体验发挥作用，我们需要*某种东西*来帮助人们理解他们所服务的原始数据。

不幸的是，任何试图创建和维护数据文档的人都知道，这说起来容易做起来难。组织拥有如此多的数据，以至于通常不可能可靠地记录哪怕是最重要的数据集。由于数据变化如此频繁，文档也经常变得陈旧，导致人们不信任文档，就像他们不信任底层数据一样。

为了解决这些问题，组织有时会雇用全职的数据管理员，他们负责定义数据文档并确保它保持最新。这不仅昂贵，而且很少奏效。数据管理员被分配到他们不是专家的部门记录数据；结果，他们最终向领域专家寻求帮助。组织可以通过雇佣更多的管家来解决这个问题，但是很多组织不能雇佣一个管家，更不用说几个了。

看到这一挑战，许多数据团队依靠良好的公民意识。数据团队的一名成员为其他人创建了一个谷歌文档，以记录所有真实的规范来源以及与之相关的*陷阱*。文档在 Slack 上发布，几个星期后，人们会努力更新和扩展它。但是好习惯却消失了。数据公民忘记更新它，或者，可以理解，忙于他们的日常工作，没有时间去做。新员工加入公司，从未接受过更新文档的培训。

事实证明，好公民身份也行不通。数据文档是一项艰巨的任务，它太大了，无法由负担过重的数据管理员来维护，而且太普遍了，无法在分散的人群中共享。

# 在博士之外思考

这些努力失败了，因为从根本上说，文档是错误的解决方案。事实证明，数据并没有向熵靠拢，而是向熵冲刺。

商业变化很快；数据源是不断发展的；对问题的回答几乎和回答一样快就过时了。试图记录这样的系统就像记录新闻一样——这只是人们了解时事的方式。(在这方面已经有过尝试，[比如 Vox](https://www.vox.com/2014/3/30/5555690/welcome-to-vox) ，它是作为一个网站推出的，网站上满是新闻“解释者”，起到一种文档的作用。这种模式从未真正坚持下来，Vox 很快成为一个在结构上与其他任何网站没有区别的新闻网站。)

数据团队应该着眼于组织大量不断变化的信息的系统，而不是着眼于解决方案的文档系统。幸运的是，有很多例子。

*   新闻网站——Vox 试图取代——并最终成为——一个标准的新闻网站。但是如果目标是让人们了解时事，传统的新闻网站做得很好。他们成功是因为他们提供了源源不断的信息。即使你没有阅读每一个故事，firehose 也会让你对当前事件有一个大致的了解，就像 Instagram 和脸书 feeds 让你被动地了解你朋友的最新消息一样。**我们能从中学到什么？**广播时间很重要。在人们面前获取信息，即使没有一次谈话能解释一切，也能让人们保持消息灵通。
*   **维基百科**——维基百科成功地“记录”了大量信息，并对时事做出了非常迅速的反应。然而，它之所以有效是因为它的规模:它是由成千上万的人维护的，这些人因为维基百科的数十亿读者而受到激励。我们能从中学到什么？“公民文件”只在大范围内有效。大约 0.05%的维基百科读者在某个月编辑了一篇文章。即使公司可以让人们以 50 倍的速度更新数据文档，一个 1000 人的公司也只有 25 个人贡献他们的文档。这远远不足以跟上数据变化的步伐。
*   Yelp—Yelp(以及其他几十个类似的网站)依靠餐馆顾客来撰写评论，而不是从专业评论家那里获取评论。因此，评论的更新比其他情况下要频繁得多。**我们能从中学到什么？**当我们谈论文档时，我们倾向于关注所有者。客户的意见也是有用的，即使它只是将一些东西标记为过时。对于非常了解该领域的“精明”客户来说尤其如此。
*   Quora 和 stack overflow——这些网站专注于回答问题，而不是记录信息。这让人们能够解决特定的需求，为重要的问题提供了一种机制，让它们浮出水面，减少了对实时文档的关注，增加了对即时答案的关注。这也鼓励人们回答问题——记录事情——因为答案的价值更加清晰。我们能从中学到什么？集中回答问题的机制可以更容易地激励人们参与，并确保他们的参与更有针对性。然而，这在答案稳定的环境中效果最好。在快速变化的领域中，时间点答案很快就会过时。
*   谷歌— 谷歌使用爬虫来索引网络，与之前存在的其他搜索引擎没有什么不同。然而，其真正的创新是 PageRank，它使用一个网站的反向链接数量作为其可信度的代理。随着时间的推移，谷歌建立了一些工具，让网站所有者能够更好地控制从他们的网站上获取哪些索引，以及如何优化他们的排名。对于企业来说，谷歌甚至开始为人们提供直接上传信息的方式。**我们能从中学到什么？**发现系统可以从自动化开始，并在不同的监管级别上分层，以便人们进行审查、纠正和扩充。
*   **互联网**——没有什么变化比互联网更快，然而，我们仍然在驾驭它方面做得很好。这是通过几种方式实现的:书签和搜索。首先，我们有我们经常去的主要网站——像 Reddit、Twitter、纽约时报等等。对于其他一切，我们通过搜索找到它，这是由流行度驱动的。我们从来没有(好吧，美国在线曾经尝试过)试图记录互联网，我们使用目录来浏览互联网。我们能从中学到什么？有时候，完全抛开“组织”，完全依靠搜索，以使用为动力，会更好。

# 经验教训

这些教训——以及我们从过去的错误中吸取的教训——将很快融入我们组织和管理数据的方法中。我们认为它们会在很多方面得到应用。

## 多回顾，少记录。

就像谷歌通过自动绘制流行地图来组织世界信息一样，我们也需要对数据做同样的事情。我们需要根据人们已经使用数据的方式，尽可能多地自动记录。目标不应该是创造一个完美的真理来源，而是一个看似真实的草图。

一旦记录下来，我们就应该把它交给数据所有者和领域专家来审查、纠正和扩充。通过这种方式，人们只需要记录属于部落知识的东西——一些东西看起来是一种方式，但实际上意味着不同的东西——而机器通过扫描数据仓库、BI 工具、Slack、JIRA 等的日志和 API 来自动化其余的事情。

## 让这里变得混乱

鉴于组织目前拥有的数据量，任何审查和记录数据的工作都必须有所侧重。Quora 和 StackOverflow 向我们展示了问题，而不是文档，有助于揭示最重要的问题。

在数据方面，我们应该遵循同样的原则:利用人们提出的问题来寻找数据热点，并将我们的精力集中在这些问题上。这意味着你的数据的一些角落会很乱，一些概念会没有记录。没关系，只要有一种方法可以识别这些区域何时“升温”

## 不仅仅是业主

通过 Yelp，我们发现精明的消费者往往比餐馆老板更能帮助其他顾客。顾客知道其他顾客关心什么，而且通常也更诚实。

同样，数据消费者知道他们信任什么数据，不信任什么数据。他们是被烧伤过的人，他们遇到了数据的*陷阱，*并以艰难的方式了解了它们。他们是您的分析师、数据科学家和业务用户。让他们在这个过程中有发言权，不要仅仅依赖数据所有者。

## 趁热吃吧

最后，我们从过去的数据文档中学到的最重要的事情是，尽管困难重重，但开始是容易的部分。维护最新的文档要困难得多。

解决这个问题的最好方法是在数据生成时记录数据。当更改已经是首要任务时，将其作为添加新事件、表或复制作业的过程的一部分。如果可能的话，把它嵌入到开发过程中，当人们不包括必要的更新时就纠缠他们。这将文档的负担转移到上游，使其成为开发周期的一部分。否则，“修复文档”会被添加到不断增长的后续项目中，而我们实际上永远也不会去做。

—

我们喜欢宣扬数据对现代组织的重要性，我们拥有的数据越多，它就越有价值。我们会提出相反的观点:组织拥有*太多*数据。如果没有更好的组织方式，大量的数据将会变得不堪重负。

幸运的是，我们正在取得进展。说到管理数据，我们已经找出了一堆行不通的东西，一些行之有效的东西，并且在相邻空间中有很好的相似之处。通过遵循这些经验教训，最重要的是，通过认识到我们需要良好的公民意识、良好的产品和良好的流程的组合，我们最终可以建立一个为所有人服务并得到所有人信任的数据社会。
